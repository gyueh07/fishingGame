# FishingLife UI 개편판

기존 텍스트 낚시게임의 저장 데이터와 게임 규칙을 유지하면서 화면과 파일 구조를 개편한 버전입니다.

## v25.1 대용량 양동이·전투 기록 저장

- 양동이는 계정 문서 한 칸에 전부 넣지 않고 `users/{닉네임}/bucketShards`의 128개 묶음으로 나누어 저장합니다.
- 물고기 9,999마리 모의 시험에서 본문 계정 기록은 약 4KB, 가장 큰 물고기 묶음은 약 45KB로 유지되었습니다.
- 물고기를 잡거나 능력치를 변경하거나 한 마리 판매할 때 실제로 달라진 묶음만 다시 저장합니다.
- 보스전과 PVP 다시보기는 `users/{닉네임}/battleRecords`에 별도로 저장하여 전투할 때마다 양동이 전체를 다시 저장하지 않습니다.
- 큰 기기 복구본은 `localStorage` 대신 용량이 큰 IndexedDB에도 보관합니다. 기기 저장 공간 부족이 클라우드 자동저장을 중단시키지 않습니다.
- 기존 계정은 새 버전으로 처음 로그인할 때 양동이와 전투 기록이 자동으로 새 저장 방식으로 옮겨집니다.
- 화면의 `저장 지연`은 인터넷 연결 문제일 때만 표시하고, 구버전은 `업데이트 필요`와 새로고침 안내로 구분합니다.

## v25.0.1 실제 시간 낚시터·수족관

- 기기의 실제 시간에 따라 낚시터가 새벽(05~07시), 낮(08~16시), 노을(17~19시), 밤(20~04시)으로 자동 전환됩니다.
- 시간대마다 하늘·바다 색뿐 아니라 해의 높이, 달, 별, 수면 반사광이 달라집니다. 낚시 확률과 전투 능력치는 바뀌지 않습니다.
- 설정에서 실제 시간 낚시터를 끄면 기존 밝은 기본 배경을 사용합니다.
- 광장의 수족관에서 양동이 물고기를 최대 5마리 전시하며 물고기를 클릭하면 기존 상세정보 화면이 열립니다.
- 수족관 물고기는 등급별 색·광륜·입자가 적용되고 영원과 공허는 전용 회전 광륜을 사용합니다.
- 전시 중인 물고기는 판매·일괄판매·전송·합성 재료에서 보호됩니다. 보스전·PVP 출전, 합성 본체, 진화에는 사용할 수 있습니다.
- 수족관은 물고기를 복사하지 않고 양동이의 고유 ID만 저장하므로 기존 능력치·체력·합성·진화 정보가 그대로 유지됩니다.
- 낚시터에는 새벽·낮 같은 시간대 이름 없이 현재 시각만 표시됩니다.
- 수족관 광장에서 다른 선장의 공개 수족관을 찾거나 구경하고, 전시 물고기의 능력치·진화·합성 정보를 읽기 전용으로 확인할 수 있습니다.
- 처음 발견한 물고기의 `NEW`는 낚은 순간에만 한 번 표시되고 양동이·최근 기록·상세정보에는 표시되지 않습니다.

## v24.9 전체 초기화·한 기기 접속

- Firestore를 비운 뒤 만들어지는 모든 계정은 Lv.1, 0원, 빈 양동이로 시작합니다.
- 브라우저를 새로 열면 이전 로그인과 로컬 게임 기록을 자동으로 불러오지 않습니다. 로그인 성공 전에는 낚시와 모든 게임 메뉴가 잠깁니다.
- 한 계정은 한 기기에서만 접속할 수 있습니다. 다른 기기에서 접속 중이면 로그인이 거부되며, 비정상 종료된 접속 잠금은 90초 뒤 풀립니다.
- 탈퇴는 경고 화면, 비밀번호 확인, 최종 확인을 거친 뒤 계정과 세션 및 백업 3칸을 함께 삭제합니다.
- 복구 화면은 `서버 기록`과 `이 기기 기록` 중 하나를 고르는 쉬운 화면으로 정리했습니다.
- v24.9.2부터 비밀번호가 맞으면 새 로그인을 허용하고 기존 접속을 즉시 종료합니다. 로그인 직후 로컬 캐시의 예전 접속 기록을 다른 기기로 오인하던 문제도 수정했습니다.
- v24.9.3부터 접속 종료 전 `users`와 `accountSessions`를 서버에서 다시 대조하여, 새 기기가 정상 접속권을 받은 직후 잘못 로그아웃되는 문제를 차단합니다.
- v24.9.4부터 게임 변경 즉시 계정별 미저장 안전본을 기기에 기록합니다. 다음 로그인에서는 같은 계정·초기화 버전·서버 기준 기록이 모두 일치할 때만 종료 직전 진행도를 자동 복구합니다.
- 전송받은 물고기는 크기·능력치·별·합성·진화·현재 체력을 유지한 채 양동이에만 들어오며, 도감 발견·발견 횟수·`NEW` 표시에 영향을 주지 않습니다. 한 마리와 다중 전송 모두 같은 규칙을 사용합니다.

## v24.8 구버전 덮어쓰기 차단

- 모든 `users/{닉네임}` 쓰기는 `writeSchemaVersion: 250`, `writeProtocol: 4`, 증가하는 `writeProtocolSeq`, 매 요청의 `writeGuardAt: serverTimestamp()`를 함께 기록합니다.
- `firestore.rules`는 새 표식을 이번 요청에서 실제로 갱신하지 않은 쓰기를 거부합니다. 구버전이 문서에 남은 버전 숫자를 그대로 물려받아도 통과할 수 없습니다.
- 낚싯대 레벨·총 낚시 횟수·누적 수익이 현재 서버 기록보다 낮아지는 저장은 클라이언트와 Firestore 규칙 양쪽에서 차단합니다. 돈과 양동이는 정상 소비·판매·전송이 있어 감소 금지 대상에서 제외합니다.
- 각 계정의 서버 쓰기 전 `gameState`를 `users/{닉네임}/backups/slot0~2`에 순환 저장합니다. 최초 v24.8 저장, UTC 날짜별 첫 저장, 전체 복구 직전, 송금·물고기 전송 직전에 생성됩니다.
- 백업에는 비밀번호와 로그인 세션 토큰을 복사하지 않습니다.
- 모든 계정에서 브라우저 종료 직전 Firebase에 저장되지 않은 기록을 dirty 표식과 마지막 정상 동기화본으로 보존합니다. 다음 로그인에서는 서버 최신 변경과 로컬 미저장 변경을 3방향으로 비교하고, 사용자가 확인하기 전 자동저장을 멈춥니다.
- 송금·물고기 전송은 일반 자동저장과 같은 순서열에서 실행합니다. 전송 중 새로 생긴 낚시·판매·강화 결과는 전송 완료 상태와 병합한 뒤 이어서 저장하며, 전송한 물고기는 양쪽에 중복되지 않습니다.
- `config/game`을 Firebase 서버에서 확인하기 전에는 로그인·회원가입·자동저장을 허용하지 않습니다. 오프라인 캐시만으로 계정을 덮어쓸 수 없습니다.
- 안전 버전 확인에 실패하면 전면 보호 화면으로 게임 진행을 잠그고, 재확인에 성공한 뒤 서버 데이터를 다시 불러옵니다.
- 계정 본문 `gameState`는 850KB 안전 제한을 유지하며, 큰 양동이와 전투 기록은 하위 컬렉션으로 분리되어 이 제한에 포함되지 않습니다.
- 실행 가능했던 구버전 `original.html`은 Firebase 코드를 전부 제거한 최신 게임 이동 페이지만 남겼으며, `validation/`은 배포 산출물에서 제외합니다.
- 탈퇴 시 비밀번호를 다시 확인하고 계정 문서, 접속 잠금, 백업 3칸을 한 번에 삭제합니다.

> 이 보호막은 예전 브라우저 파일의 **우발적인 덮어쓰기와 진행도 하락**을 막습니다. 현재 게임은 Firebase Authentication 없이 공개 클라이언트가 `users` 문서를 직접 다루는 구조라, 코드를 고의로 변조한 공격까지 막는 인증 장치는 아닙니다. 계정 소유권·치팅 방지는 Firebase Auth와 서버 함수로 송금·물고기 전송·복구를 옮기는 다음 마이그레이션이 필요합니다.

### 반드시 지킬 배포 순서

1. Firebase Console의 Firestore 규칙 탭에 포함된 새 `firestore.rules`를 **먼저 게시**합니다. CLI라면 `firebase deploy --only firestore:rules --project fishinggame-1c8ac`입니다.
2. 바로 이어서 v25.1의 `index.html`, `css/`, `js/`, `original.html`을 호스팅에 배포합니다. Firebase Hosting이면 `firebase deploy --only hosting --project fishinggame-1c8ac`입니다.
3. `config/game` 문서에는 숫자 `minimumBuild: 250`, 숫자 `minSaveSchemaVersion: 250`, 숫자 `minimumWriteProtocol: 4`, 불리언 `maintenance: false`를 설정합니다.
4. 기존 테스트 계정으로 로그인한 뒤 물고기 수, 보스전 기록, 송금과 물고기 전송을 확인합니다. 첫 로그인에서 기존 양동이가 자동 이전됩니다.
5. 규칙과 호스팅 배포가 모두 끝나기 전에 이용자가 게임을 진행하지 않도록 점검 시간을 두고, 완료 후 페이지를 새로고침하도록 안내합니다.

`firebase.json`에는 Hosting과 Firestore 규칙이 함께 정의되어 있습니다. **이번 전체 초기화 배포에서는 `firebase deploy`를 단독으로 실행하지 말고**, 위의 두 `--only` 명령을 규칙 → Hosting 순서로 연속해서 실행하세요.

규칙 게시 직후 이미 열려 있던 구버전 탭은 저장이 거부됩니다. 모든 이용자에게 **기존 탭을 닫고 v25.1 주소를 새로 열도록** 공지하세요.

## v24.7 차희승 계정 긴급 복구

- 이 버전은 `차희승` 계정에 한해서 Firebase의 현재 기록과 접속한 브라우저에 남아 있는 게임 기록을 로그인 직후 비교합니다.
- 복구 후보를 고르기 전에는 자동저장·실시간 동기화를 멈춰, 남아 있던 브라우저 기록이 되돌아간 서버 기록으로 덮이지 않게 합니다.
- 확인 기준은 **낚싯대 Lv.1,200 이상·PVP 총전투력 약 5.3억**입니다. 레벨뿐 아니라 지갑, 물고기 수, 총 낚시 횟수, 고등급 물고기도 함께 확인하세요.
- 맞는 후보를 찾은 브라우저에서만 `이 브라우저 기록으로 전체 복구`를 누르세요. 복구 직전 Firebase 기록은 해당 브라우저에 별도 안전본으로 보관됩니다.
- 비교 중 서버 데이터가 바뀌면 송금·선물·다른 기기의 진행을 보호하기 위해 복구를 중단하고 최신 서버값과 다시 비교합니다.
- 인스타그램·카카오톡·일반 브라우저는 저장공간이 서로 다를 수 있습니다. 기존 기록이 남아 있을 가능성이 있는 각 접속 경로를 같은 서비스 주소에서 확인하고, 확인 전에는 앱 데이터·브라우저 캐시를 삭제하지 마세요.
- 어떤 접속 경로에도 기준과 맞는 후보가 없다면 이 클라이언트만으로 과거 지갑·물고기를 정확히 재구성할 수 없습니다. 그 경우 Firebase 백업·내보내기·관리자 스냅샷이 필요합니다.

## 이번 버전에서 바뀐 점

- 1대1 재생과 이전 전투 기록에서 `PVP_SIDE:left/right` 내부 표식이 노출되지 않고 보는 사람 기준 아군·적으로 표시
- 새 1대1 프레임에는 내부 표식을 저장하지 않아 기록·로그·다시보기에서 같은 문제가 재발하지 않도록 변경
- 받은 소식 도착, 메시지·송금·선물, 프리셋·잠금·강화 등 짧은 결과를 판매 결과처럼 아이콘·제목·설명으로 나뉜 게임 알림 카드로 통일
- 보스 다중 공격을 물고기 한 마리씩 순서대로 재생하고 각 대상의 피해 전후 체력·감소량·감소율을 별도 결과 카드로 표시
- 치명타·회피 전용 배지와 화면 효과, 수르트 검 파괴·피닉스 부활·보스 각성용 BOSS PHASE 카드 추가
- 랭킹을 보유 골드·낚싯대 레벨·PVP 전투력 3개 선택 카드로 나누고 선택한 순위만 1열로 표시
- 보스 난이도 상금의 부동소수점 오차를 제거하고 실제 지급값을 10원 단위로 반올림
- 공허 물고기 5종을 검정·진홍, 백색·청록, 진홍·금색, 보라·청록, 보라·분홍·청록의 서로 다른 전용 스킬 UI로 분리
- 1대1 스킬 재생을 `스킬명·설명 → 체력 변화 결과` 두 단계로 나눠 스킬 소개 중에는 체력바가 먼저 줄지 않도록 변경
- 보스전 상태 영역에 영구 삭제된 스킬, 관측 중인 스킬과 남은 턴, 마침표 문장 0~5 게이지 표시
- 전용 공허 색상은 라이브 전투 화면에만 적용하고 아래 전투 로그의 기존 물고기 등급·보스 색상은 유지
- 공허 무별 공격력·체력을 영원 ★★와 같은 90만~210만·1,350만~3,150만 구간으로 상향하고 별 1~3개도 동일한 3·7·20배 규칙 적용
- 기존 공허 물고기의 별 구간 내 위치와 현재 체력 비율을 보존하면서 합성 누적 전이·진화 기준 수치까지 새 공허 범위로 자동 변환
- 20개 보스의 전투당 1회 크레이지 궁극기를 즉시 피해·행동 제어·지속 효과가 결합된 최종기 수준으로 강화
- 바하무트·티아마트·요르문간드·앙그라 마이뉴·에레보스·아자토스에 크레이지에서만 발동하는 신규 패시브 추가

- 훈련소 구간별 누적 성장: Lv.1~5 단계당 +20%, Lv.6~10 +50%, Lv.11~15 +100%
- 기존 양동이 물고기에도 상향된 훈련소 배율 즉시 적용
- PERFECT, GREAT, GOOD, BAD 낚시 타이밍 미니게임과 확률 보정
- 타이밍 구간을 노랑·초록·파랑·빨강의 단색으로 구분하고 판정은 낚아채는 순간에만 표시
- GREAT는 기존 낚싯대 확률, PERFECT는 소폭 보너스, GOOD·BAD는 단계별 페널티 적용
- BAD는 일반·희귀 비중이 크게 증가하고 초월 이상 확률과 도주 성공률에 강한 페널티 적용
- 최근 획득한 물고기 10마리 표시 및 클릭 상세정보
- 양동이의 180마리 표시 제한을 제거해 보유 물고기 전체 표시
- 양동이 카드와 상세정보에 기존 공격력·체력·회피·치명타·치명타 피해 1~3별 표시 복원
- 양동이 체력 칸에 현재 체력/최대 체력, 잔여 퍼센트와 체력바 표시
- 양동이에서 회복 중인 물고기의 남은 회복시간을 실시간 표시
- 양동이·보스 파티·전투 결과에서 기절 상태를 일반 회복과 다른 색과 문구로 표시
- 보스전 재생에서도 양동이와 동일한 물고기별 이모지 사용
- 물고기 도주 시 낚시 상태를 즉시 종료하고 이름·등급·판정·도주 확률을 전용 화면과 문구로 표시
- 낚시 등급 추첨과 확률 표시를 동일한 100% 정규화 함수로 통합
- 낚시터를 다시 열 때 이전 획득 연출이 반복되던 문제 수정
- 물고기 이름에 맞는 전용 이모지 표시
- 기존 1~1750레벨 성능을 유지한 낚싯대 2000레벨 확장
- 1750~2000레벨에 공허 보스 5종의 코어 재료 연결
- 채팅형 명령어 화면을 제거하고 시세·도감·보스전·PVP를 게임형 화면으로 표시

## 파일 구조

```text
index.html                 화면 구조
original.html              구버전 직접 접속 차단·최신 게임 이동
firebase.json              Firebase Hosting 캐시·구버전 경로 차단 설정
firestore.rules            구버전 users 쓰기·진행도 감소·계정 삭제 차단
css/legacy.css             기존 로그·모달 기본 스타일
css/game-ui.css            새 반응형 게임 UI
js/data/fish-data.js       물고기, 등급, 크기, 특성 데이터
js/data/combat-data.js     전투 범위, 보스, 코어 데이터
js/core.js                 계정, 저장, 공통 상태, 연구·훈련
js/combat.js               보스전과 PVP 전투
js/gameplay.js             낚시, 경제, 양동이, 도감 등
js/commands.js             기존 명령어 연결과 시작 처리
js/ui.js                   새 화면 표시와 버튼 연결
```

## 배포

프로젝트의 파일과 폴더 구조를 유지한 채 정적 웹 호스팅의 루트에 업로드합니다. 기존 게임과 같은 Firebase 프로젝트를 사용하므로 Firebase 설정과 허용 도메인은 기존 배포 환경을 그대로 사용합니다. `validation/`은 검증용 원본이라 프로덕션에 업로드하지 않습니다. Firebase Hosting을 사용하면 포함된 `firebase.json`이 HTML 캐시 금지와 `/original.html`, `/validation/**` 리다이렉트를 함께 적용합니다.

## 저장 데이터 호환

전투 능력치 버전은 원본과 동일하게 유지했습니다. 저장된 물고기의 `combat` 데이터가 다시 생성되거나 약해지지 않습니다.
